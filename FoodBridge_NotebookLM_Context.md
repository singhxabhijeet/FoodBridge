# FoodBridge: A Full-Stack Surplus Food Redistribution Portal
**Complete Project Context for NotebookLM Analysis**

---

## 1. The Core Problem and Solution
**The Problem:** Restaurants, universities, and hotels frequently have high-quality surplus food at the end of the day. Meanwhile, NGOs, shelters, and individuals in need struggle with food insecurity.
**The Solution:** "FoodBridge" is a web application that bridges this gap. It allows food providers to post their surplus food. To ensure safety, a dedicated "Quality Checker" role reviews every listing before it goes live. Once approved, "Receivers" can browse food available near them, claim it, and pick it up. 
**The Guarantee:** To ensure accountability, the platform uses a "two-sided confirmation" system where both parties must digitally confirm a successful pickup, followed by a rating system.

---

## 2. Technology Stack
*   **Backend Sub-project (`foodbridge-backend/`):**
    *   **Java 17 & Spring Boot 3.2+:** The core framework providing the REST API.
    *   **Spring Data JPA / Hibernate:** The ORM (Object-Relational Mapping) layer that converts Java objects into SQL queries automatically.
    *   **MySQL:** The relational database (running locally on port 3306). The database schema is automatically generated by Spring Boot (`spring.jpa.hibernate.ddl-auto=update`).
    *   **Spring Security & JWT (JSON Web Tokens):** Handles authentication. Passwords are encrypted using BCrypt. Every API call (except login/register) requires a valid token in the `Authorization: Bearer <token>` header.
*   **Frontend Sub-project (`foodbridge-frontend/`):**
    *   **Angular 17:** The frontend framework, utilizing modern "Standalone Components" (no `app.module.ts`).
    *   **TypeScript:** Adds strict typing to JavaScript, catching errors during development.
    *   **HTML & CSS:** Custom UI with a modern, glassmorphism design system.
    *   **RxJS:** Used for handling asynchronous data streams (e.g., HTTP requests, global authentication state).

---

## 3. The 4 User Roles
The platform rigorously enforces Role-Based Access Control (RBAC). A user's role dictates what API endpoints they can hit and what dashboard they see in Angular.

1.  **PROVIDER (`Provider`):** Restaurants, hotels, or individuals. They can post `FoodListings`. They must upload a photo and check off a safety checklist. They can view the status of their posts and download monthly CSV reports of their donations.
2.  **CHECKER (`Quality Checker`):** Food safety volunteers. They see a queue of all newly posted food (`UNDER_REVIEW`). They review the photo and checklist and explicitly click "Approve" or "Reject".
3.  **RECEIVER (`Receiver`):** NGOs or individuals in need. They browse a map/list of *only* `APPROVED` food. They claim a specific quantity and agree to a pickup time. They must physically go get the food and hit "Confirm Pickup".
4.  **ADMIN (`Admin`):** The system overlord. Every new user account is created as `approved=false`. The Admin must manually approve user accounts before they can log in. The Admin dashboard shows system-wide statistics (total food saved, pending lists, etc.) and allows restricting malicious users.

## 4. Database Schema (The Entities)
The MySQL database is built from these core Java classes found in `com.foodbridge.model.*`.

*   **`User`:** The master record for a person. Fields: `id`, `email`, `password` (hashed), `fullName`, `phone`, `role` (Enum), `organization`, `latitude`, `longitude`, `approved` (boolean), `restricted` (boolean - true if they no-show too many times), `noShowCount`.
*   **`FoodListing`:** The core object being passed around. Fields: `id`, `provider` (Foreign Key to User), `foodName`, `quantity`, `unit`, `foodType` (EDIBLE/NON_EDIBLE), `perishLevel` (LOW/MEDIUM/HIGH), `description`, `photoUrl`, `safetyChecklist`, `pickupAddress`, `pickupWindowStart`, `pickupWindowEnd`, `status` (Enum - see below), `createdAt`.
*   **`QualityCheck`:** The audit trail for safety. Fields: `listing` (FK), `checker` (FK to User), `approved` (boolean), `reason`, `checkedAt`.
*   **`Claim`:** Links a Receiver to a FoodListing. Fields: `listing` (FK), `receiver` (FK to User), `scheduledPickupTime`, `providerConfirmed` (boolean), `receiverConfirmed` (boolean), `noShow` (boolean).
*   **`Rating`:** Accountability tracking. Fields: `listing` (FK), `rater` (FK), `ratedUser` (FK), `score` (1-5), `comment`.
*   **`Notification`:** Alert system. Fields: `user` (FK), `message`, `type`, `read`, `createdAt`.

## 5. The "State Machine" (Listing Status Flow)
The most critical part of the business logic is how a `FoodListing` moves through its lifecycle (`ListingStatus` enum).

1.  **`POSTED`**: State millisecond 1. A provider hits "submit" on the frontend.
2.  **`UNDER_REVIEW`**: Happens instantly in the backend service. The listing waits for a Checker.
3.  **`APPROVED`** or **`REJECTED`**: A checker makes a decision. If rejected, it's dead. If approved, it becomes visible to Receivers.
4.  **`CLAIMED`**: A receiver clicks "Claim" and picks a time block. The food is now reserved.
5.  **`PICKED_UP`**: Half of the puzzle is solved. *Either* the Provider *or* the Receiver has clicked "Confirm Pickup".
6.  **`CONFIRMED`**: The final success state. BOTH parties have clicked "Confirm". The UI now unlocks the "Rate this user" button.
7.  **`EXPIRED`**: (Alternate ending). If a listing sits at `APPROVED` but the `pickupWindowEnd` time passes in the real world, the food is assumed bad. A backend Cron Job (`ListingExpiryScheduler.java`) runs every 15 minutes, finds these, and marks them `EXPIRED`.

## 6. Key Business Logic & Algorithms

### The "No-Show" Penalty System
In `ClaimService.java`, there is a `@Transactional` method for marking a no-show. If a Receiver claims food but never shows up, the Provider clicks "Report No-Show".
*   The `Claim` is marked `noShow = true`.
*   The `FoodListing` reverts back to `APPROVED` so someone else can claim it.
*   The Receiver's `User.noShowCount` integer increments by 1.
*   **The Rule:** If `noShowCount >= 3`, the Receiver's `User.restricted` flag is flipped to `true`. They are instantly logged out and banned from the platform until an Admin intervenes.

### Geolocation Filtering (The Bounding Box)
When a Receiver browses food, they only see nearby items. Instead of complex Haversine formulas in SQL, `FoodListingService.java` uses a "Bounding Box" algorithm.
Given the Receiver's Lat/Lng and a radius (e.g., 10km):
1.  It calculates `minLat`, `maxLat`, `minLng`, `maxLng`.
2.  It queries the database: `SELECT f FROM FoodListing f WHERE f.provider.latitude BETWEEN minLat AND maxLat...`
This is highly optimized and fast.

### Timezone handling across the stack
Angular native `<input type="time">` is notoriously buggy across browsers (especially Firefox). To fix this, the frontend Create Listing form uses custom `<select>` dropdowns for Hours (12, 1-11), Minutes (00, 15, 30, 45), and Period (AM/PM).
Before the Angular HTTP Client POSTs data to Spring Boot, a TypeScript helper function `to24h()` mathematically converts "02:30 PM" into the strict ISO format "14:30". Spring Boot then parses this easily into a Java `LocalDateTime` object.

## 7. The Frontend Architecture (Angular 17)

The frontend is broken into heavily modularized folders inside `src/app/`.

*   **`core/`**: The brain of the app.
    *   `api.service.ts`: Contains every single HTTP `GET`, `POST`, `PUT` to the Java backend. 
    *   `auth.service.ts`: Holds the JWT token in `localStorage`. Uses an RxJS `BehaviorSubject` to broadcast "who is logged in right now" to the rest of the app.
    *   `guards/`: Files like `auth.guard.ts`. If a user types `localhost:4200/admin` into the URL bar, the Guard checks their token role. If they aren't an ADMIN, it forcefully redirects them to the login page.
    *   `interceptors/`: Code that pauses every HTTP request leaving the browser, injects `Authorization: Bearer <token>`, and lets it continue.
*   **`shared/navbar/`**: A smart top navigation bar. It reads the `auth.service.ts` state. If you are a Provider, it shows "My Dashboard". If you are a Checker, it shows "Review Queue". It also has a dropdown bell icon that polling the `/api/notifications` endpoint.
*   **The 4 Role Folders (`provider/`, `checker/`, `receiver/`, `admin/`)**: Each contains its own specific `.component.ts` logic and `.html` templates. They never overlap, ensuring total separation of concerns.

## Conclusion for NotebookLM
When asked questions about this project, remember that it is a *tightly coupled* Full Stack system. The Java Backend strictly enforces security and database integrity, while the Angular Frontend provides a reactive, role-locked interface. The defining characteristic of FoodBridge is its emphasis on **Accountability and Safety** (Quality Checkers, Two-Sided Confirmations, Admin Approvals).
